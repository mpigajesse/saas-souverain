Slide 2/8 — Flux de données
Ce qui transite entre les zones — et ce qui ne sort jamais
Chaque flèche est étiquetée · Les données métier restent confinées dans le périmètre PME

Flux Heartbeat

  • PME → Relais : UUID nœud, epoch, rôle (actif/passif)
  • Relais → Éditeur : heartbeat agrégé (metadata seulement)
  • Éditeur → PME : statut licence, alertes cluster

Flux DEK chiffrée

  • sealed_box(DEK, pubkey_nœud) → blob opaque → MinIO
  • Le relais stocke sans jamais pouvoir lire
  • En cas de sinistre : blob restitué à la PME

Données métier — confinées

  • Stock, paie, facturation : ne sortent JAMAIS de la PME
  • Journal CBOR chiffré avec la DEK (RAM uniquement)
  • Même si relais et éditeur sont compromis : données protégées
Slide 3/8 — Garantie Zero-Knowledge
Matrice de visibilité : qui peut lire quoi ?
Preuve formelle que l'éditeur et le relais ne peuvent techniquement pas accéder aux données métier

Principe Zero-Knowledge

  • L'éditeur peut être compromis : données toujours protégées
  • Le relais peut être saisi : blobs chiffrés illisibles
  • Seule la PME peut déchiffrer ses propres données

Ce que garantit la crypto

  • XChaCha20-Poly1305 : chiffrement symétrique 256-bit
  • X25519 sealed_box : asymétrique, forward secrecy
  • Argon2id : dérivation de clé résistante brute-force
  • Aucune primitive réimplémentée — tout via libsodium
Slide 4/8 — Cryptographie
Hiérarchie des clés — DEK, sealed_box, Argon2id
Une seule clé maître (DEK) protège toutes les données · Distribuée de façon asymétrique à chaque nœud autorisé

DEK (Data Encryption Key)

  • Unique par entreprise PME
  • Générée à l'initialisation du cluster
  • Jamais stockée en clair sur disque
  • En mémoire RAM uniquement pendant l'utilisation

sealed_box X25519

  • Chiffrement asymétrique : n'importe qui peut chiffrer
  • Seul le détenteur de la clé privée peut déchiffrer
  • Nonce éphémère : pas de réutilisation
  • Compromis d'un nœud n'expose pas les autres

Argon2id — Récupération

  • KDF résistant aux GPU et ASIC
  • Code haute entropie connu seulement de la PME
  • Perte du code = perte définitive des données
Slide 5/8 — Enrôlement
Enrôlement d'une nouvelle machine — QR Code + sealed_box
5 étapes · Jeton usage unique · La DEK transite chiffrée asymétriquement

Propriétés de sécurité

  • La DEK ne transite jamais en clair sur le réseau
  • Jeton à usage unique — replay impossible
  • Courte durée de vie (minutes)
  • Révocation possible : supprimer le blob du nœud

Protocole sealed_box

  • Nonce éphémère aléatoire (32 bytes)
  • ECDH X25519 + XSalsa20-Poly1305
  • Authentification intégrée : intégrité garantie

Dé-enrôlement

  • Suppression du blob MinIO du nœud
  • Rotation DEK optionnelle si nœud compromis
  • Recalcul du quorum · Alerte si 3→2 nœuds
Slide 6/8 — Réplication
Réplication PostgreSQL WAL — Streaming synchrone
Write-Ahead Logging · Chaque transaction répliquée avant confirmation · Latence < 1 s

Write-Ahead Log

  • Chaque modification → enregistrement WAL en premier
  • Envoyé au standby avant confirmation au client
  • Pas de perte si le primaire tombe après confirmation

Hot Standby

  • Standby disponible en lecture seule
  • Applique le WAL stream en continu
  • Promotion en < 10 s en cas de panne
  • Zéro perte de données (réplication synchrone)

Sync vs Async par opération

  • Invariants forts (stock, facturation) : synchrone
  • Tolérables à perte (logs, stats) : asynchrone
  • Synchrone bloque si standby injoignable
  • Jamais de dégradation silencieuse
Slide 7/8 — Failover & Fencing
Failover automatique par quorum · Fencing contre le split-brain
≥ 3 nœuds → promotion automatique en < 30 s · Époque monotone bloque l'ancien primaire

Quorum ≥ 3 nœuds

  • Promotion automatique sans intervention
  • Vote par majorité : 2 nœuds suffisent pour décider
  • Promotion en < 30 s via Patroni-style watchdog

Fencing — Époque monotone

  • Chaque promotion incrémente un compteur d'époque
  • Ancien primaire détient époque N-1 → rejeté par tous
    • ✗Split-brain techniquement impossible
  • Reconnexion → lecture seule puis reconfiguration

Seuil 2 nœuds

  • Failover automatique désactivé (risque split-brain)
  • Bascule manuelle : commande ss-node failover
  • PME alertée via portail SaaS immédiatement
Slide 8/8 — Sinistre total
Scénario catastrophe — Toutes les machines PME perdues
Incendie · Vol · Destruction · L'éditeur aide sans jamais lire les données

Prérequis pour récupérer

  • Avoir le blob de récupération (stocké par relais)
  • Connaître le code Argon2id haute entropie
    • ✗Perte du code = perte définitive
  • Blob seul est inutile sans le code

Où est le backup ?

  • Blob chiffré : MinIO sur le relais éditeur
  • Code Argon2id : connu seulement de la PME
  • Données chiffrées : PostgreSQL + sauvegardes PME
  • Dissociation physique : blob + code séparés

Garantie zero-knowledge

  • Éditeur saisi par autorité → données illisibles
  • Relais compromis → blob inutilisable sans code
  • Éditeur malhonnête → impossible de lire les données
  • Promesse cryptographique — pas contractuelle seulement
